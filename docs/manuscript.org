#+title: Kugire Data Curation for the Kokinwakashu TEI/XML Data

* Introduction

Kugire, breaks within waka poems, are a basic structural element of waka. However, data on kugire is not well-developed. In particular, when incorporating kugire data for the TEI/XML data of waka, standardized, publicly available data on kugire is lacking. It is necessary to create annotations independently.

This project attempts to implement kugire as an interpretive annotation layer on top of the TEI/XML data of the Kokinwakashu, which is the one of the most important waka anthologies. Rather than embedding kugire directly into the waka text, maintaining the kugire as an annotation layer allows for the retention of information about what criteria were used for the annotation, and whether alternative interpretations exist.
The data of kugire can be applicable to the analysis of the structure of waka literature, enjambment in waka, and can be used as a case study for the development of ambiguity-aware, invisibly layered interpretive annotations in literary and linguistic studies.

* Previous works

** Kugire-related data

As mentioned above, data on kugire is not well-developed.  [atuko315] is one of the few data sources of kugire, which provides a program to identify kugire and provides kugire data of Hyakuninisshu for evaluation. The program is also performed on the Kokinwakashu and the Shinkokinwakashu to compare the results with [okaasa] and [kamihiro]. However, kugire, different from the physical text data, is an interpretive layer without a single correct answer. Thus, the dataset is expected to not be a fixed, authoritative, definitive annotation provided by a single source. Instead, the dataset is expected to be a collection of annotations from multiple sources with clear documentation of their criteria and interpretations.

** Kugire standard

[okaasa] proposes a standard for kugire. He defines the criteria for determining kugire as follows:

1. The necessary condition :: The existence of two or more semantically indenpendent core sentences in the poem;
2. The sufficient condition :: The existence of a clear grammartical break in the poem.  

While both conditions remain interpretive ambiguty. Considering the interpretive nature of kugire, instead of making a single, fixed annotation of kugire, we maintain its ambiguity and descriptively annotate the kugire from grammatical and semantic perspectives from multiple sources.
That is, implementing kugire as an annotation layer on top of the TEI data. That allows for the retention of information about whose judgement it is, what criteria were used, and whether alternative interpretations exist.

* Materials

** Base data

The base data for this study is a lexically extended version [chin2026] of the Karoku second year manuscript TEI data of the Kokinwakashu [ikuura]. [ikuura] provides detailed descriptions of the physical format of the manuscript, various annotations, e.g., marginalia, apparatus, notes, and a list of authors with their social attributes. The data allows for the further annotation and serch of the poem text.  [chin2026] map a lexical and morphological layer on top of [ikuura] based on the Hachidaishu Vocabulary Dataset [Hodoscek2022].
The mapped data provides lexical information and the segmentation of the waka text, which is the basis for the kugire annotation. On this mapped data, we add the kugire annotation.

** Reference data

For the grammatical kugire, we refer to the Hachidaishu Part-of-Speech Dataset [Yamamoto], which provides conjugation data for each verb and part of speech information. These information functions as a reference for the grammartical kugire criteria.

For the semantic kugire, we refer to the explanatory translations of the Kokinwakashu by [kaneko], which is the earlist contemporary translation in the 20th century.
The translation data is open-sourced and available on Zenodo [yamamoto].

* Annotation pipeline

The annotation follows a Large Language Model (LLM) assisted pipeline of the following steps:

1. Rule-based automatic suggestion of candidate kugire positions based on the [Yamamoto]
2. LLM-based automatic suggestion of candidate kugire positions based on the translation data of [kaneko,Yamamoto]
3. Review and correction on the poped-up editor, which shows the original poem and it reference data.
4. Rendering the reviewed annotation into the TEI data.

The pipeline is for fastening the annoation process. The one-poem-one-batch-one-buffer-at-a-time review and correction is for cognitive ease and accuracy.

** Rule-based grammar-wise kugire suggestion

The above-mentioned rule-based program by [atuko315] use the conjugation information to identify candidate kugire positions. The program identifies candidate positions based on the following criteria:

1. Phrases ending with verbs in the first suffix group and the third suffix group, in gakko bunpou, which correspond to the termination form, the imperative form, and the realis form.
2. Phrases ending with the copula, e.g., ``nari", ``tari".
3. Phrases ending with sentence-final particles,e.g., particles expressing question, prohibition, excalamation.

Our automatic suggestion of candidate kugire positions is similar to that, but we allow for multiple candidate positions and devide candidates into strong, miderate,
and weak ones.

Strong candidates are based on the termination form of tokens. For example, in the first poem of the Kokinwakasu below, the past tense auxiliary verb ``keri" is in the termination form, which is a strong indication of a grammatical break after it, making it a strong candidate for kugire.

#+begin_example
年の内に / 春はきにけり / 一とせを / こぞとやいはん / ことしとやいはん

tosi=no uchi=ni / paru=pa k-in-iker-i / pito+tose=wo / kozo=to=ya ip-am-u / ko+tosi=to=ya ip-am-u

year=gen inside=dat / spring=top come-mnf-hps-ccl / one+year=acc / last.year=cmp=xcl say-cjt-ccl
/ this+year=cmp=xcl say-cjt-ccl

translation: Within the year / spring has come! / one year / shall (we) call (it) last year? / shall (we) call (it) this year? 
#+end_example

Moderate candidates are based on the copula and information structure, which possibly indicate a break but not certainly. For example, in the second poem of the Kokinwakashu below, segments before and after the token ``wo" are dependency-wise connected, but the former segments present a scene, while the latter segments present a conjectural statement regarding the scene, which can be an indication of a break.

#+begin_example
袖ひちて / むすびし水の / こほれるを / 春立つけふの / 風やとくらん

sode pit-ite / musub-isi mizu=no / kopor-er-u=wo / paru tat-u kepu=no / kaze=ya tok-uram-u

sleeve be.wet-seq / scoop.up-pst.adn water=gen / freeze-sta-adn.nl=acc / spring start-adn today=gen / wind=xcl melt-cjt-ccl

translation: making sleeve wet, / water (I) scooped up / has frozen, while / spring starts, starting today's / wind! would (you) melt (it)? 
#+end_example

Weak candidates involve variant semantic interpretations, where sentence-final particles may indicate a break but semantically the former and latter part are interdependent. For example, in the third poem of the Kokinwakashu below, the sentence-final particle of question is used in the second segment, which can be an indication of a break, but the following segment is an answer to the question, which shows a semantic interdependence between the two segments, making the break less certain.

#+begin_example
春霞 / たてるやいづこ / みよしのの / 吉野の山に / 雪はふりつつ

paru+kasumi / tat-er-u=ya izuko / mi-Yosino=no / Yosino=no yama=ni / yuki=pa pur-itutu

spring+haze / rise-sta-ccl=xcl where / hpf-Yosino=gen / Yosino=gen mountain=dat / snow=top fall-itr

translation: Spring haze / where is (it) rising /  Yoshino / on the Mountain Yoshino / snow is falling
#+end_example

The source of judgement is recorded as cached log for each poem, which is used for the review and correction step. In the future, the source of judgement can be recorded as a separate element in the TEI data, which allows for the retention.

The process of the rule-based suggestion takes the original poem and its morphologicaland lexical data by [yamamoto] as arguments, and outputs the candidate kugire positions.

** LLM-based explanation-wise kugire suggestion

Besides the rule-based suggestion, we also use a large language model (LLM) to suggest candidate kugire positions based on the explanatory translation data of [kaneko] encoded by [yamamoto]. While the rule-based approach derives candidate positions from morphological and grammatical features, the LLM-based approach derives them from the semantic structure of the modern Japanese prose translation, offering a complementary perspective that can capture breaks rooted in meaning rather than grammar alone.

The LLM employed is =qwen2.5=, a locally hosted model served via Ollama [ollama]. Running the model locally avoids external API dependencies and ensures reproducibility. The model is prompted with the five segments of the original poem and the corresponding explanatory translation, and asked to output candidate kugire positions in a structured JSON format: ={"positions": [...]}=.

To guide the model toward a consistent and interpretively grounded judgment, the prompt instructs it to follow a three-step reasoning process:

1. *Segment alignment*: Identify which segments of the original poem correspond to which parts of the explanatory translation.
2. *Break detection*: Locate sentence break points in the translation — places where the meaning comes to a close, where an exclamation ends, or where a semantic shift occurs.
3. *Position mapping*: Map the identified break points back to the corresponding segment boundaries in the original poem.

This three-step process mirrors the judgment workflow that a human annotator would follow when using an explanatory translation as a reference for kugire determination.

The prompt also includes three few-shot examples drawn from real Kokinwakashu poems, each demonstrating the same three-step reasoning chain: one example with no kugire, one with a second-segment break, and one with a fourth-segment break. The few-shot examples use a generic label (「現代語訳:」) rather than naming a specific translator, making the prompt reusable with any future translation source without modification.

The output of the LLM is parsed as JSON, and valid segment indices (1-based, excluding the final segment) are converted to kugire positions tagged with the translator code as their evidence source. Invalid or out-of-range positions are silently discarded. Each position produced by this process is recorded as a separate =<k>= element in the TEI data with =source= set to the translator code (e.g., =kaneko=), independent of any positions produced by the rule-based method.

The LLM-based suggestion is computationally heavier than the rule-based approach, requiring approximately 30 seconds per poem. For the full Kokinwakashu of 1111 poems, a batch run takes several hours. Results are cached as plain-text draft files (one per poem per source), so the suggestion step need not be repeated unless the model or prompt changes.
